# 아키텍처 가이드

> 프로덕션 코드를 작성·수정하는 모든 에이전트(programmer, refactorer)가 따르는 구조/레이어링 원칙.

## 레이어링
- **presentation**: 입력 검증, 직렬화, 프레임워크 어댑터. 도메인 로직 금지.
- **application**: 유스케이스 오케스트레이션. 트랜잭션 경계.
- **domain**: 핵심 비즈니스 규칙. 프레임워크/인프라 의존 금지.
- **infrastructure**: DB, 외부 API, 메시징 등 구현체.

의존성 방향은 항상 **바깥 → 안쪽** (presentation → application → domain).
domain 은 어떤 바깥 레이어도 import 하지 않는다.

## 모듈 경계
- 한 모듈이 다른 모듈의 내부 구현을 직접 참조하지 않는다.
- 공유가 필요하면 명시적인 public interface 를 통해서만 노출한다.

## 의존성 주입
- 생성자 주입을 기본으로 한다.
- 전역 싱글톤, 서비스 로케이터 패턴 지양.

## 데이터 흐름
- 도메인 객체와 DTO 를 분리한다. 경계에서 변환한다.
- 불변 데이터를 우선한다.

## 새로운 파일을 만들기 전에
1. 기존 파일에 추가할 수 있는지 먼저 확인한다.
2. 레이어가 불명확하면 plan.md 의 해당 커밋 설명을 다시 읽는다.
3. 그래도 불명확하면 가장 보수적인 위치(기존 구조 따라가기)를 택하고 보고한다.
